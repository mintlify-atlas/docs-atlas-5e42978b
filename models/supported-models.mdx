dflash-mlx supports any model that has a matching DFlash draft on HuggingFace. The runtime auto-resolves the draft from a built-in registry, so most users only need to pass the target model reference. If no registry entry exists for your model, you can supply a draft explicitly with `--draft`.

## Tested target/draft pairs

These pairs are validated end-to-end on Apple Silicon with the DFlash runtime:

| Target | Draft |
|--------|-------|
| `Qwen/Qwen3.5-4B` | `z-lab/Qwen3.5-4B-DFlash` |
| `Qwen/Qwen3.5-9B` | `z-lab/Qwen3.5-9B-DFlash` |
| `mlx-community/Qwen3.5-27B-4bit` | `z-lab/Qwen3.5-27B-DFlash` |

## Architecture compatibility

<CardGroup cols={2}>
  <Card title="Qwen3.5 (hybrid GatedDeltaNet)" icon="star">
    Qwen3.5 models use a hybrid architecture combining GatedDeltaNet recurrent layers with standard attention layers. dflash-mlx is optimized for this architecture and activates two additional precision mechanisms:

    - **Tape-replay rollback** — records an innovation tape during verification and replays only accepted steps via a custom Metal kernel, keeping GatedDeltaNet state coherent without snapshotting the full recurrent state.
    - **JIT SDPA 2-pass** — long-context verification (`N >= 1024`) uses a custom Metal attention kernel that stays numerically aligned with stock MLX attention.
  </Card>
  <Card title="Qwen3 (pure attention)" icon="check">
    Qwen3 models use a standard transformer architecture with no recurrent layers. dflash-mlx supports these models and the speculative decoding loop runs correctly, but the tape-replay rollback and SDPA precision kernels are not activated because there are no GatedDeltaNet layers to manage.
  </Card>
</CardGroup>

<Accordion title="How the runtime detects architecture">
  At load time, `detect_target_family()` in `runtime.py` inspects the target model's layers for `linear_attn` or `is_linear` attributes. If any layer has these, the model is classified as `"hybrid_gdn"` and the speculative hooks (tape recording, SDPA chunking) are installed. Otherwise it is classified as `"pure_attention"` and the hooks are skipped.
</Accordion>

## Compatibility requirements

To use a model with dflash-mlx:

1. A DFlash draft model must exist on HuggingFace (either in the registry or provided via `--draft`).
2. The draft model must be a `dflash_qwen3` model type loadable by `DFlashDraftModel`.
3. The target model must be loadable by `mlx_lm` (MLX-format weights).

## Using an unsupported model

If your target model is not in the built-in registry, pass a compatible DFlash draft explicitly:

```bash
dflash --model your-org/your-model \
       --draft z-lab/compatible-draft \
       --prompt "Your prompt here"
```

The same flag works with `dflash-serve`:

```bash
dflash-serve --model your-org/your-model \
             --draft z-lab/compatible-draft \
             --port 8000
```

<Warning>
  Models without a matching DFlash draft will be rejected at startup. If no `--draft` flag is provided and no registry entry is found for your target model, the runtime raises a `ValueError` listing supported base models and pointing to `https://huggingface.co/z-lab` for available drafts.
</Warning>
